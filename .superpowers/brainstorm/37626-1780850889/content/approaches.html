考试成绩查询 — 三种交互方案对比

目标：查询大学期间全部成绩，操作越简单越好

背景发现

你的 scheduleApp 已有完整的成绩功能代码（ScoreScreen、ScoreViewModel、ScoreParser、Room 实体），但当前 ScoreViewModel 用 8 次 POST 请求逐学期拉取，效率低且容易失败。而 DirectLoginViewModel 中的导入流程已经证明：一次 GET 请求（year=&term=&para=0）就能拿到全部成绩。

三种方案

A

年份 + 学期下拉框 + 查询按钮

参考 GlutAssistant 方案

优点

  • 用户可按需查询单个学期
  • 每次请求数据量小，加载快
  • 传统的教务系统风格，用户熟悉

缺点

  • 看全部成绩需要多次切换 + 多次查询
  • 年份列表需要硬编码或推算
  • 每次查询都要等待网络请求
  • 用户不知道哪学期有成绩，可能空查询
B

年份 + 学期下拉框（含"全部"）+ 本地过滤

参考 glut (Android) 方案

优点

  • 一次请求拉取全部成绩，后续过滤无延迟
  • "全部"选项让用户一键看所有成绩
  • 切换年份/学期即时响应（纯本地过滤）
  • 离线也可查看已缓存的成绩

缺点

  • 首次加载全部数据可能稍慢
  • 下拉框仍占屏幕空间
  • UI 略显"传统"，不够现代
C

一键获取全部 + 按学期自动分组 + 可选筛选

推荐方案 — 更现代的移动端体验

优点

  • 零操作门槛：点击刷新即获取全部成绩
  • 按学期自动分组展示，一目了然
  • 顶部可选轻量筛选（年份/学期 chip 或底部 sheet）
  • 每个学期区块显示平均绩点和总学分
  • 与你现有的 ScoreScreen 设计方向一致

缺点

  • 数据量大时列表较长（但按学期折叠可缓解）
  • 需要设计筛选 UI（但可以后续迭代）
